# Registration spawn-rate scan — 2026-07-13

## What this is

Follow-up to `baseline-2026-07-11.md`'s flagged known-limitation: under a
device cold-start burst, `dg-sat`'s single-writer SQLite serializes
first-contact writes, producing long-tail request latency. That baseline
only ever tested one spawn rate (80/s); this experiment scans across
several to find where the tail actually starts becoming a problem, rather
than reasoning from a single datapoint.

**Method:** fixed device count (2000), steady-state traffic only
(`make locust-steady-state` — `/device`+`/config`+`/events`, update flow
and admin traffic excluded via `--exclude-tags update`, no seeding), fresh
`dg-sat` instance + fresh data volume per run. Spawn rate (`-r`) varied
across 10/20/40/80/160 users/sec. Run time set generously past each rate's
expected ramp-up completion (200s/100s/50s/25s/12.5s respectively) so no
run was truncated mid-tail; full device registration (`Link` header's
`last` offset = 1999) confirmed after every run.

Command shape (rate=40 shown; only `-r`/`-t` change between runs):

```
make setup NUM_DEVICES=2000
# bring up dg-sat, then:
docker compose run --rm --no-deps -e LOCUST_HEADLESS=true locust locust \
  -f locustfile.py PerfUser --exclude-tags update \
  --host https://dg-sat:8443 --headless \
  -u 2000 -r 40 -t 75s \
  --csv /data/ratescan-r40 --num-devices 2000
```

Raw per-rate stats CSVs are not included alongside this doc (five short
headless runs, no seeding, no HTML report requested) — the numbers below
are the complete summary; re-run the command above with the shown `-r`/`-t`
to reproduce any row.

## Results

| Spawn rate | Run time | Requests | Failures | Median | p95 | p99 | **Max** | Max vs. previous rate |
|---|---|---|---|---|---|---|---|---|
| 10/s | 300s | 892,643 | 0 | 10ms | 22ms | 47ms | **741ms** | — |
| 20/s | 150s | 438,949 | 0 | 10ms | 23ms | 52ms | **1.44s** | 1.95× |
| 40/s | 75s | 219,470 | 0 | 10ms | 24ms | 62ms | **6.92s** | 4.80× |
| 80/s | 60s | 169,253 | 0 | 10ms | 26ms | 220ms | **26.5s** | 3.83× |
| 160/s | 90s | 257,101 | 0 | 10ms | 24ms | 61ms | **39.8s** | 1.50× |

All five runs: **0 failures**, all 2000 devices confirmed registered.
Aggregate steady-state throughput stayed flat across every rate
(~2850-2980 req/s) — see "What this says" below for why that's expected
and doesn't contradict the max-latency growth.

## What this says about dg-sat

- **Median/p95 latency is essentially rate-independent.** 10ms median and
  ~22-26ms p95 hold at every spawn rate tested, including 160/s. The
  *typical* request is unaffected by how fast devices are arriving — only
  a small, growing minority of requests (the ones that land exactly during
  the SQLite single-writer's busy window) pay the contention cost.
- **The tail has a clear knee between 20/s and 40/s.** Growth from
  10→20/s is roughly linear (1.95×, in line with the 2× rate increase
  itself). Growth from 20→40/s is a 4.8× jump — well past what a linear
  relationship would predict — and 40→80/s continues nonlinearly (3.83×).
  By 80→160/s the ratio drops back to 1.5×, suggesting the tail is
  approaching (not yet at) a ceiling near ~40s at this device count,
  rather than growing unboundedly.
- **This is a ramp-up-only effect, not reduced steady-state capacity.**
  Aggregate throughput across the whole run is flat (~2850-2980 req/s)
  regardless of spawn rate — confirming the latency growth is confined to
  the burst window while devices are still registering, not a change in
  what the server can sustain once ramp-up completes.
- **Still zero failures at every rate tested, including 160/s** — `5s`
  `_busy_timeout` retries plus Locust's own default timeouts absorbed the
  full tail without a single dropped/errored request in any of the five
  runs. The p99→max gap (e.g. 220ms → 26.5s at 80/s) says this is a small
  number of very unlucky requests, not a broad slowdown.

## Caveats / what this doesn't answer

- **Only tested at 2000 devices**, not the 5000 used in the main
  baselines — chosen so five runs at increasing rates stayed fast to
  iterate. The absolute tail lengths above likely don't transfer directly
  to 5000 devices; the *shape* (flat median, nonlinear tail with a knee
  around 20-40/s) is the transferable finding, not the exact millisecond
  values.
- **Single run per rate, no repeats.** The `SQLITE_BUSY` retry pattern
  this measures is inherently about scheduling luck (which requests happen
  to land during the writer's busy window) — a single sample per rate
  could be somewhat noisy, especially at the higher rates. Treat the exact
  "1.50× at 160/s" figure as directional, not precise.
- **Doesn't test rates between 20 and 80** (e.g. 30, 50, 60) — the knee is
  bracketed to "somewhere in 20-40" but not pinpointed further. A finer
  scan in that specific range would sharpen this if it matters for a real
  decision (e.g. picking a production spawn-rate cap).
- **Steady-state-only scenario** (update flow and admin traffic excluded)
  — deliberately, to isolate the registration-burst signal, but a real
  device fleet's first-contact traffic wouldn't be this clean. The main
  baselines' mixed-scenario numbers remain the more realistic end-to-end
  picture; this doc answers a narrower question in isolation.

## Bottom line

If you need a rule of thumb for this server's SQLite-backed writer under a
cold-start burst: **spawn rates up to ~20/s keep tail latency in the
sub-1.5s range; above ~40/s the tail grows sharply and lands in the
10-40s range**, though it still resolves with zero failures. The existing
`Makefile` default of 80/s sits well past the knee this scan found —
consistent with `README.md`'s existing note that the cap exists because
higher rates serialize on the single SQLite writer and produce long-tail
latency, and that raising it further "without checking error rates is not
recommended." This scan adds a number to that warning: 80/s is already in
the regime where tail latency is measured in tens of seconds, not a
regime this data suggests is close to safe headroom.
