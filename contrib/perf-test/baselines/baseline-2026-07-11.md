# Baseline run — 2026-07-11

## What this is

A full-scale (5000 simulated devices) headless run of `contrib/perf-test`
against a fresh `dg-sat` instance, with the TUF fixture seeded
(`SEED_UPDATE=1`) and the default mixed task set (steady-state device
check-ins, the update-check/download flow, and admin device listing all
running together — no `--tags` scenario filter applied). Kept as a
reference point for future comparisons, replacing the `2026-07-10` baseline
now that the harness has moved on (task tagging, named `make locust-*`
scenario targets, profile/scene composition — see `README.md`).

Command used (see `README.md` for flag meanings):

```
make setup NUM_DEVICES=5000 SEED_UPDATE=1 UPDATE_TAG=main UPDATE_NAME=perf-target-1
# bring up dg-sat, then:
docker compose run --rm --no-deps -e LOCUST_HEADLESS=true locust locust \
  -f locustfile.py --host https://dg-sat:8443 --headless \
  -u 5000 -r 80 -t 5m \
  --csv /data/baseline-2026-07-11 --html /data/baseline-2026-07-11.html \
  --num-devices 5000
```

Run window: 2026-07-11 12:50:52Z (setup start) → 12:57:17Z (locust
shutdown). `NUM_ADMINS` at its `Makefile` default (1).

Files alongside this doc:
- `baseline-2026-07-11.html` — full Locust HTML report (charts + tables)
- `baseline-2026-07-11_stats.csv` — final per-endpoint stats
- `baseline-2026-07-11_stats_history.csv` — time series (per-second snapshots)
- `baseline-2026-07-11_failures.csv` / `_exceptions.csv` — both empty (0 rows of data)

## What this says about dg-sat

Same two-profile split observed in the prior baseline, essentially
unchanged (the tagging/scenario-selection work added in between did not
touch runtime task-selection *within* a default/untagged run, or the
server-side write path — so this is expected, not a new finding):

1. **Write-touching endpoints** (`/device`, `/config`, `/events`) have a
   healthy 13-14ms median but a long tail: 95th percentile 1.9-3.7 seconds,
   max ~240s on all three — dg-sat's single-writer SQLite
   (`storage/db.go`: `db.SetMaxOpenConns(1)`, `_busy_timeout=5000`) under a
   5000-device cold-start burst. Still 0 failures — a latency problem, not
   correctness.
2. **Read-only endpoints** (`/repo/*`, `/ostree/*`) have a tight
   distribution: 12ms median, 25ms p95, ~240ms max on the four siblings
   that structurally can't be a device's first contact.
3. **`GET /repo/timestamp.json`** is again the one read-only-looking
   endpoint with a long tail (95th %ile still 25ms, but max 237.8s) — same
   **CONFIRMED** mechanism as the prior baseline (see "Anomalies worth
   flagging" below): it's always step one of `UpdateFlow`'s
   `SequentialTaskSet`, so it's the only read-only endpoint structurally
   eligible to carry a device's first-ever `checkinDevice` write.
4. **`GET /v1/devices`** (admin) stays tight at 222ms max, consistent with
   the prior baseline's 232ms — a pure read with no `CheckIn`/
   `DeviceCreate` write dependency, unaffected by the registration burst,
   exactly as expected.

## Anomalies worth flagging

- **Traffic volume is heavily skewed toward the update-check/download
  endpoints** (~150k requests each) relative to the three steady-state
  device tasks (~9k-23k each) and the admin endpoint (~5.9k), even though
  `PerfUser.tasks` weights `get_device` at 5, `get_config` at 2,
  `post_events` at 3, and `UpdateFlow` at only 1. **CONFIRMED** in the prior
  baseline's investigation (see `baseline-2026-07-10.md` for the full
  reproduction) and unchanged here: Locust's nested-`TaskSet` execution
  never returns control to the parent once entered
  (`SequentialTaskSet.run()` is an unconditional `while True`), so the
  ~1-in-11 of `PerfUser` instances that roll `UpdateFlow` early are
  permanently absorbed into it for the rest of the run. If this weighting
  imbalance is unwanted for a *mixed* run like this one, run the isolated
  scenarios instead (`make locust-steady-state`, `make locust-update`,
  `make locust-admin` — see `README.md`), which sidestep the absorption
  behavior entirely by tag-filtering before the run starts rather than
  relying on runtime random selection.
- **`/repo/timestamp.json`'s solitary long tail among the `UpdateFlow`
  siblings** — **CONFIRMED**, same mechanism as the prior baseline:
  `checkinDevice`'s `CheckIn()` write fires on any mTLS endpoint when a
  device's real header values first differ from its seeded/blank state,
  and `timestamp.json` is always `UpdateFlow`'s first step, so it's the
  only read-only endpoint that can structurally carry that cost. See
  `baseline-2026-07-10.md`'s "Anomalies worth flagging" section for the
  full live-traced reproduction (device-index tracing across a 300-device
  burst); not independently re-traced for this run since the underlying
  code path (`checkinDevice`/`CheckIn`) is unchanged since that
  investigation.
- ~~`GET /v1/devices`'s max latency (221.8s) is new and unexplained~~ — this
  was a units error in the original version of this document, not a real
  finding. See the correction note immediately below.
- Everything else (0 failures, 0 exceptions, full 5000/5000 registration,
  tight latency on the four non-`timestamp.json` `UpdateFlow` endpoints,
  and `/v1/devices` staying tight throughout) looks healthy and consistent
  with the `2026-07-10` baseline.

**Correction (2026-07-11, later):** this section originally flagged
`GET /v1/devices`'s max latency as "221.8s and unexplained," with a
speculative guess about SQLite read/write queuing. That was a units error
in this document, not a server finding — the raw stats CSV reports max
response time in **milliseconds**; `221.75531298853457` is 221.76ms
(0.22s), not 221.8 seconds. Re-checked directly against
`baseline-2026-07-11_stats.csv`: `/v1/devices` sits in the same tight
200-240ms cluster as `/ostree/config`, `/repo/targets.json`, etc. — no
anomaly, no investigation needed. The "Headline numbers" table and point 4
above have been corrected to 222ms.

**Second correction (2026-07-11, caught by independent review):** the
first correction above claimed to have checked "every endpoint's raw max
value the same way," but that check was incomplete — it only fixed the
`/v1/devices` row. Four more rows in the "Headline numbers" table had the
identical ms-labeled-as-s bug: `/repo/snapshot.json` (240.5ms, not 240.5s),
`/repo/targets.json` (216.8ms, not 216.8s), `/ostree/download-urls`
(240.6ms, not 240.6s), and `/ostree/config` (240.2ms, not 240.2s). All four
already had the correct millisecond figures in this doc's own prose (point
2 under "What this says about dg-sat" always said "~220ms max" for these
same endpoints) — only the table's unit suffix was wrong. Table corrected
now; no other numbers in this document were affected (the five
write-touching/first-contact rows — `/device`, `/config`, `/events`,
`/repo/timestamp.json`, and the `Aggregated` row — are genuinely in the
100k+ ms range and correctly shown in seconds).
